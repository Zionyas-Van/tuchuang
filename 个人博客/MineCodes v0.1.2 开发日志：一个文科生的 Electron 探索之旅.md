# MineCodes v0.3.0 开发日志：一个文科生的 Electron 探索之旅

> **MineCodes** 是一个为 Minecraft 玩家准备的辅助工具 —— 查指令、找物品 ID、存笔记，一站式搞定。


---

## 前言

先简单交代一下背景：我是文科生，代码不算我的专业领域。

《MineCodes》从最初由 AI 裸生成jian只能翻指令列表的简陋版本，到今天能够查物品 ID、做笔记、支持深色模式的完整工具，经历了不少推倒重来的时刻。今天这篇日志想和你聊聊最近的开发进展，也顺便梳理一下走过的弯路和教训。

**温馨提醒**：如果你在用的是旧版本，建议直接下载最新版安装包，因为在正式版发布之前的每一个版本都可能完全不一样。

## 最近更新

### 指令笔记：从“看一眼就忘”到“随手记录”

以前我自己用 MineCodes 的时候，经常遇到一个尴尬的情况：查到一个好用的复杂指令，当时看懂了，下次用的时候又忘了参数顺序。反复查同一个指令确实烦人。

所以这次更新加了一个 **指令笔记** 功能。主界面右侧就是笔记输入区，你可以在查阅指令的同时，顺手把自己的理解或者简化写法记录下来：

![指令笔记截图](https://raw.githubusercontent.com/Zionyas-Van/stackedit-app-data/master/imgs/2026-05-03/ZWa6rJaOAHDT9PAj.png)

笔记保存在本地，重启软件也不会丢。其实就是 JSON 文件，后面如果你愿意，也可以手动编辑。

### 物品 ID 查询：找物品不用再切网页

`/give` 指令写起来最烦的就是记物品 ID。以前我自己的习惯是浏览器里一直开着 Minecraft Wiki 的物品列表页，切来切去真的很打断思路。

现在内置了一个物品 ID 查询模块：

![物品ID查询截图](https://raw.githubusercontent.com/Zionyas-Van/stackedit-app-data/master/imgs/2026-05-03/BJAIrji1OuepzwgJ.png)

搜索框输入中文名或英文名都能匹配，点击直接复制 ID，顺手就能拿去写指令。目前覆盖了 Java 版的主流物品，后续会持续补充。

### 深色模式：眼睛少受罪

这大概是用户反馈最多的需求 —— 之前一直是亮色界面，晚上用确实晃眼。新版本终于做了深色模式适配，支持跟随系统主题，也可以手动切换：

![深色模式截图](https://raw.githubusercontent.com/Zionyas-Van/stackedit-app-data/master/imgs/2026-05-03/LUn6ACHhex06kRVv.png)

## 开发日志：三个核心功能的实现思路

### 功能 1：指令库 + 收藏系统

这个功能其实是最早做的，也是整个工具的基础。Minecraft 的指令体系说大不大、说小不小，简单指令就那么十几个，但加上参数组合和各种变体之后，能翻出几十种用法。

最初的设计就是简单列一个表格，后来觉得这种展示方式太“文档化”了，不够直观。所以改成了卡片式布局，每个指令单独展示，点展开可以看到详细的用法说明和示例。

收藏功能的实现其实很简单 —— 就是一个本地存储的 ID 数组，收藏的指令会单独列在“收藏夹”视图里。但难的是如何让用户一眼就知道哪个指令被收藏了，以及在收藏夹列表怎么方便地取消收藏。前后改了三次交互逻辑，现在的方案是：每个卡片右上角有一个爱心图标，点击即收藏/取消，状态实时同步。

![收藏示例](https://raw.githubusercontent.com/Zionyas-Van/stackedit-app-data/master/imgs/2026-05-01/8gajfHUZhf76POrS.png)

### 功能 2：物品 ID 查询 —— 数据源怎么来的？

物品 ID 数据是一个大工程。Minecraft 从 1.12 到 1.20 版本之间，命名空间和数据格式都变了好几轮。为了确保数据的准确性，我主要用了三个来源：

1. **Minecraft Wiki** 上公开的物品列表作为基准；
2. 官方 JSON 数据文件中扒取的注册名；
3. 自己抽空对照着一个个验证。

最终录入的条目覆盖了原版生存模式中会用到的绝大多数物品，接近 800 条。当然了，不排除有遗漏，如果遇到找不到的物品 ID，欢迎在 GitHub 提 Issue 告诉我。

### 功能 3：笔记模块 —— 最简单的实现往往最可靠

笔记模块的开发是这轮更新里最顺畅的一个环节。不是因为简单，而是因为一开始就很清楚要做一个什么样的东西：

- 不需要云端同步（太复杂，而且用户未必需要）；
- 不需要富文本（能用 Markdown 就够了）；
- 不需要多笔记本分类（那是另外一个层面的需求）。

最终实现了一个纯本地的 Markdown 编辑器，所见即所得谈不上，但基本的标题、列表、代码块都支持，足够做指令笔记用了。数据存在 `localStorage` 里，退出自动保存。

也许以后会加导出/导入功能，但短期内不打算过度复杂化。

## 为什么用 Electron？

这个问题其实挺多人问过 —— “一个小工具为什么要用 Electron 包个壳？直接网页不好吗？”

我的考虑有几个：

1. **跨平台**：Minecraft 玩家什么系统的都有，Electron 生成一套代码覆盖 Windows、macOS（ARM64 / x64）、Linux 三端，省事。
2. **本地优先**：笔记数据、收藏记录都存本地，不依赖服务器，用户不用担心隐私问题。
3. **熟悉的前端技术栈**：虽然我是文科生，但 HTML / CSS / JS 相对好上手，Electron 的学习曲线比从头学 C# 或 Swift 要友好得多。

当然，Electron 的缺点也很明显 —— 包体积大（光是 chromium 内核就占几十 MB）、内存占用高。但对于一个偶尔打开查一查指令的工具来说，这些缺点在可接受范围内。

## 技术踩坑记录

### 坑 1：跨平台打包的路径问题

Electron Builder 在 Windows 和 macOS 上的资源路径策略不太一样。一开始没注意这个问题，结果打包出来的 macOS 版本死活读不到本地 JSON 数据文件。

排查了两天才发现 —— 开发环境下 `__dirname` 指向的是项目根目录，但打包后在 macOS 上，资源文件被打包到了 `Resources/app.asar` 里面，需要通过 `process.resourcesPath` 来访问。

解决方案：封装了一个 `getResourcePath()` 函数，判断环境返回不同路径。血的教训，写 Electron 应用一定不要在代码里写死任何绝对路径。

### 坑 2：深色模式切换时的渲染闪烁

深色模式实现的时候遇到了一个很烦的问题：手动切换主题时，页面会有明显的闪烁，看起来像是先渲染亮色再变成深色，体验很差。

根源是主题切换 CSS 类的方式是直接操作 `document.documentElement`，但页面上的某些组件已经按旧的类完成了计算和渲染。解决办法是用 `prefers-color-scheme` 媒体查询作为默认主题，同时给主题切换加了一个 `transition` 过渡，虽然不能完全消除闪烁，但至少不会那么突兀。

### 坑 3：物品 ID 搜索的性能优化

物品列表有接近 800 条数据，输入框添加搜索事件监听之后，每次键盘敲击都触发一次全量过滤，低配机器上明显能感觉到卡顿。

优化方案很简单 —— **防抖 + 虚拟滚动**：
- 防抖：用户停止输入 150ms 后再执行过滤，减少重复计算；
- 虚拟滚动：只渲染当前可视区域的 15-20 条结果，其他先不渲染，大幅降低 DOM 操作开销。

改完之后搜索体验顺滑了很多。

## 写在最后

这个项目对我个人来说，最大的意义可能不在于做出多厉害的工具，而是提供了一个用代码解决问题的机会 —— 一边学一边写，写了再改，改了再打包，打包完发现还有问题，继续回来改。

说实话，这个过程挺折磨的。但每次看着屏幕上跑起一个自己写的新功能，那种成就感确实是打游戏刷副本给不了的。

MineCodes 会持续更新。接下来几个版本的计划包括：

- 指令自动补全和参数提示；
- 支持更多 Minecraft 版本的数据切换；
- 笔记模块增加导出为 Markdown / PDF 功能。
